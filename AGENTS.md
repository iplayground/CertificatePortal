# AGENTS.md

## 目的

本儲存庫將用於實作 iPlayground 完訓證明系統。

目前產品範圍如下：

1. 產生可供會眾下載的完訓證明。
2. 提供公開的證明驗證服務。
3. 提供管理者上傳完訓名單的作業流程。

本專案目前從空白儲存庫開始。除非後續有明確決策取代本文件，未來所有 agent 都應以本文件作為基準工作約定。

## 主要技術棧

- 執行環境：Azure Functions v4
- 語言：Python
- 部署方案：Azure Functions Flex Consumption
- 資料庫：Azure Cosmos DB
- 物件儲存：Azure Blob Storage
- 機密管理：Azure Key Vault 或 Azure 管理的應用程式設定
- 監控：Azure Monitor 與 Application Insights

## 架構基線

- 使用 Azure Functions 承載應用程式邏輯。
- 使用 Cosmos DB 儲存中繼資料、流程狀態、驗證紀錄、上傳批次與可稽核的領域資料。
- 使用 Blob Storage 儲存來源上傳檔、證書樣板與產生後的證書檔案。
- 不要將產生後的 PDF 或其他二進位檔案存入 Cosmos DB。
- 若證書生成時間不確定，或可能超出一般 HTTP 延遲預期，應將其視為非同步流程。
- 上傳與證書生成工作應優先採用事件驅動處理。

## 工作前提

- 預設正式環境部署目標為 Linux 上的 Flex Consumption。
- 本專案目前以 Python 3.13 作為基準執行版本。
- 依 Azure Functions 官方文件，Python 3.13 目前為 GA，可作為正式環境版本。
- 系統設計應保有足夠模組化，讓管理端、公開驗證端與生成流程可以獨立演進。

## 領域模型預期

至少應規劃以下邏輯實體：

- 會員
- 課程或方案
- 完訓紀錄
- 證書
- 驗證 Token
- 上傳批次
- 上傳列處理結果
- 管理者稽核事件

建議的儲存邊界如下：

- Blob Storage：
  - 上傳的 CSV 或 Excel 檔案
  - 證書樣板
  - 產生後的 PDF 檔案
- Cosmos DB：
  - 正規化後的中繼資料
  - 驗證狀態
  - 批次處理狀態
  - 證書下載資格
  - 稽核事件

## API 設計規則

- 公開 API 與管理 API 必須清楚分離。
- 所有 HTTP API 從一開始就應進行版本化，例如 `/api/v1/...`。
- 公開驗證端點必須為唯讀。
- 管理端點必須要求驗證與授權。
- 公開端點不得暴露不必要的個人資料。
- 對於可能發生重試的操作，端點應盡可能設計為冪等。

## 安全規則

- 不得將任何機密、金鑰、連線字串或 SAS Token 提交到儲存庫。
- 不得將任何 Azure 環境專屬設定、匯出的設定檔或部署憑證提交到儲存庫，例如 `local.settings.json`、publish profile、Portal 匯出的設定內容或其他含實際 Azure 設定值的檔案；僅可提交去敏後的範本與 IaC 定義。
- 只要 Azure 能乾淨支援，就優先使用 Managed Identity，而非長效機密。
- 產生 Blob Storage 下載連結時，優先使用 user delegation SAS，而非 account key 簽署的 SAS。
- 公開驗證回應應保持最小揭露並兼顧隱私。
- 除非有文件化的營運需求，否則避免記錄完整的個人可識別資訊。
- 如實務可行，應在日誌中遮罩或雜湊化敏感識別資訊。

## Function 設計規則

- 每個 function 應聚焦於單一責任。
- 共用商業邏輯應放在 trigger handler 之外。
- 處理上傳或生成工作的 functions 必須具備可安全重試的特性。
- 所有外部邊界都應進行明確驗證。
- HTTP API 應回傳可預期、機器可讀的錯誤格式。
- 若需要長時間執行的流程協調，優先使用 Durable Functions，而不是臨時拼湊的輪詢邏輯。

## 程式碼品質標準

- Python 程式碼應使用型別註記。
- 優先採用職責清楚的小型模組。
- 針對領域規則、解析、驗證與驗證邏輯撰寫測試。
- 不要把商業規則藏在框架接線程式中。
- 只有在程式意圖不夠明顯時才加入註解。
- 優先使用清楚命名，而非炫技式抽象設計。

## 建議的儲存庫結構

在建立專案骨架時，除非有更好的理由，優先採用以下結構：

```text
infra/
  bicep/
src/
  functions/
  shared/
tests/
docs/
scripts/
```

建議職責如下：

- `infra/`：基礎設施即程式碼
- `src/functions/`：Azure Functions 進入點與 trigger/binding 定義
- `src/shared/`：共用領域邏輯、服務、模型與介接層
- `tests/`：單元測試與整合測試
- `docs/`：架構說明、ADR、API 說明與維運文件
- `scripts/`：本機設定、資料工具與開發自動化腳本

## 基礎設施指引

- 因為目標平台明確是 Azure-first，初期基礎設施即程式碼優先使用 Bicep。
- 基礎設施定義必須可審查。
- 資源定義與實際部署參數應分離。
- 不應依賴手動 Portal 設定作為主要配置來源。

## 資料與流程指引

- 上傳匯入流程應先驗證，再持久化正規化狀態，最後觸發下游生成工作。
- 證書發行必須先建立可持久保存的紀錄，再讓檔案進入可下載狀態。
- 驗證流程應讀取權威中繼資料，而不是信任用戶端傳入的宣稱內容。
- 即使第一版沒有完整 UI，資料模型層仍應支援撤銷或失效處理。

## 文件要求

- 重大技術決策變更時，必須同步更新本檔案。
- 重要架構變更應記錄於 `docs/adr/`。
- 凡是 Azure 資源建立、部署、設定變更或環境初始化流程，都必須同步記錄於 `docs/`。
- 一旦引入環境變數、Storage 容器與 Cosmos 容器，應補上文件。
- 公開或管理 API 一旦實作，應提供 API 範例。

## 決策優先順序

當取捨不明確時，依下列順序優先考量：

1. 安全與隱私
2. 正確性與可稽核性
3. 維運簡單性
4. 成本效率
5. 開發便利性

- 若 `7075` 的重啟失敗，後續修正問題時必須一併處理到服務可重新在 `7075` 成功啟動，不可只停留在程式碼修正。
